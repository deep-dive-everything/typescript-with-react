# 1. 자바스크립트의 런타임과 타입스크립트의 런타임

## 1.1 런타임과 컴파일타임

- 컴파일타임 : 소스코드가 컴파일러에 의해 컴퓨터가 인식할 수 있는 기계어 코드로 변환되어 실행할 수 있는 프로그램이 되는 과정.
- 런타임 : 컴파일 과정을 마친 응용 프로그램이 사용자에 의해 실행되는 과정.

## 1.2 자바스크립트 런타임

- 자바스크립트가 실행되는 환경
- 인터넷 브라우저, Node.js 등
- 구성요소 : 자바스키립트 엔진, 웹 API, 콜백 큐, 이벤트 루프, 렌더 큐
- 런타임에서 자바스크립트 에러가 발생하므로 애플리케이션 실행중 에러가 발생함.

## 1.3 타입스크립트의 컴파일

- tsc라는 컴파일러를 통해 자바스크립트 코드로 변환.
- 고수준언어 ⇒ 고수준 언어 이므로 source-to-source compiler라고 지칭
- .ts확장자 파일을 컴파일 한 후 .js확장자의 자바스크립트코드로 만듬.
- 실행 과정
    1. 타입스크립트 소스코드를 타입스크립트 AST로 만듬 (tsc)
    2. 타입 검사기가 AST를 확인하여 타입 확인 (tsc)
    3. 타입스크립트 AST를 자바스크립트 소스로 변환 (tsc)
    4. 자바스크립트 소스코드를 자바스크립트 AST로 만듬 (런타임)
    5. AST가 바이트 코드로 변환 (런타임)
    6. 런타임에서 5번이 평가되어 프로그램 실행 (런타임)
- 타입스크립트소스의 타입은 1, 2번에서만 사용
- 작성한 타입은 타입확인에만 사용되며 최종 프로그램에 영향을 주지 않음.
- 정적 타입 검사기 : 컴파일 타임에 타입을 검사해 에러가 발생하면 프로그램이 실행되지 않음.

# 2. 타입스크립트 컴파일러의 동작

## 2.1 코드 검사기

- 컴파일 타임에 문법 에러와 타입에러를 모두 검출.
- 문법 오류 : 자바스크립트 작성시 에러가 발생하지 않는코드를 타입스크립트로 작성하면 실행전 에러가 발생하며 표시되는 에러메시지도 다름. 런타임 에러 방지
- 타입 오류 : 컴파일 타임에 타입 에러 발견. 이것이 수정되어야 타입스크립트AST를 자바스크립트 코드로 변환함.

## 2.2 코드 변환기

- tsc는 타입스크립트 코드를 각자의 런타임 환경에서 동작할 수 있도록 구버전의 자바스크립트로 트랜스파일함.
- 타입스크립트는 브라우저와 같은 런타임에서 실행할 수 없음. 자바스크립트 코드로 변환해야함.
- 트랜스파일이 완료되면 자바스크립트 파일에서 타입 정보는 지워짐
- tsc는 타입 오류가 있더라도 일단 컴파일 함. 자바스크립트로 변환과 타입 검사는 독립적으로 진행되기때문. 이때 런타임 오류로 처리하게 됨.
- tsc와 Babel의 차이점 : ES5이하의 자바스크립트 코드로 컴파일해준다는 점에서 동일하지만 바벨은 타입검사를 하지 않는다.

# 3. 타입스크립트 컴파일러의 구조

- 타입스크립트 컴파일러의 실행 과정
    - 스캐너 : .ts토큰화
    - 파셔 : 토큰 기반 AST 생성
    - 바인더 : AST 노드 기반 심볼 생성
    - 체커 : AST + 심볼 기반 타입 검사
    - 이미터 : AST + 코드 검사 기반 .js 생성

## 3.1 프로그램

- 타입스크립트 컴파일러는 tsc 명령어로 실행.
- 컴파일러는 tsconfig.json에 명시된 컴파일 옵션을 기반으로 컴파일을 수행함
- 전체적인 컴파일 과정을 관리하는 프로그램 객체(인스턴스) 생성됨.
- 이 프로그램 객체는 컴파일 할 타입스크립트 소스 파일과 소스 파일 내에서 임포트된 파일을 불러오는데, 가장 최초로 불러온 파일을 기준으로 컴파일 과정이 시작됨.

## 3.2 스캐너

- 타입스크립트 소스를 자바스크립트 소스로 변환하는 첫 번째 단계
- 타입스크립트 소스 파일을 어휘적으로 분석하여 토큰을 생성.

## 3.3 파서

- 토큰 정보를 이용하여 AST 생성.
- AST : 컴파일러가 동작하는데 핵심 기반이 되는 자료구조. 트리형태로 표현.
- 각각의 노드는 코드상의 위치, 구문 종류, 코드 내용가 같은 정보를 담음

## 3.4 바인더

- 체커 단계에서 타입 검사를 할 수 있도록 기반을 마련하는 것.
- 타입검사를 위해 심볼이라는 데이터 구조를 생성.
- 심볼에 AST에서 선언된 타입의 노드 정보를 저장함.
- 심볼을 생성하고 해당 심볼과 그에 대응하는 AST노드를 연결함.

## 3.5 체커와 이미터

- 체커
    - 파서가 생성한 AST와 바인더가 생성한 심볼을 활용하여 타입검사 수행
    - AST의 노드를 탐색하면서 심볼 정보를 불러와 주어진 소스 파일에 대해 타입 검사를 진행
    - 타입을 검증하고 타입 에러에 대한 메시지 저장
- 이미터
    - 타입스크립트 소스를 자바스크립트 파일(js)과 선언 파일(d.ts)로 생성
    - 변환 과정에서 개발자가 설정한 타입스크립트 설정 파일을 읽어오고 체커를 통해 타입 검증정보 가져옴. emitter.ts소스 내부의 emitFiles() 함수를 사용해 타입스크립트 소스 변환 진행

## 타입스크립트 컴파일 과정

1. tsc 명령어를 실행하여 프로그램 객체가 컴파일 과정을 시작함
2. 스캐너는 소스 파일을 토큰 단위로 분리
3. 파서는 토큰을 이용해 AST를 생성함.
4. 바인더는 AST의 각 노드에 대응하는 심볼을 생성함. 심볼은 선언된 타입의 노드 정보를 담음
5. 체커는 AST를 탐색하면서 심볼 정보를 활용하여 타입 검사 수행
6. 타입 검사 결과 에러가 없다면 이미터를 사용해서 자바스크립트 소스 파일로 변환
